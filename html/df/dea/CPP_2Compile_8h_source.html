12// We structure the compiler for generating C++ versions of Zeek script
13// bodies maily as a single large class. While we divide the compiler's
14// functionality into a number of groups (see below), these interact with
15// one another, and in particular with various member variables, enough
16// so that it's not clear there's benefit to further splitting the
17// functionality into multiple classes. (Some splitting has already been done
18// for more self-contained functionality, resulting in the CPPTracker class
19// and initialization information in InitsInfo.{h,cc} and RuntimeInits.{h,cc}.)
20//
21// Most aspects of translating to C++ have a straightforward nature.
22// We can turn many Zeek script statements directly into the C++ that's
23// used by the interpreter for the corresponding Exec()/DoExec() methods.
24// This often holds for Zeek expressions, too, though some of them require
25// considerations (e.g., error handling) that require introducing helper
26// functions to maintain the property that a Zeek script expression translates
27// to a C++ expression. That property (i.e., not needing to turn Zeek
28// expressions into multiple C++ statements) simplifies code generation
29// considerably. It also means that the compiler should *not* run on
30// transformed ASTs such as produced by the Reducer class. We instead
31// seek to let the C++ compiler (meaning clang or g++, for example)
32// find optimization opportunities, including inlining.
33//
34// For some Zeek scripting types, we use their natural C++ counterparts,
35// such as "zeek_uint_t" for "count" values. In the source code these
36// are referred to as "native" types. Other types, like tables, keep
37// their interpreter-equivalent type (e.g., TableVal). These are dealt
38// with almost entirely using IntrusivePtr's. The few exceptions (i.e.,
39// direct uses of "new") are in contexts where the memory management
40// is clearly already addressed.
41//
42// The user specifies generation of C++ using "-O gen-C++", which produces
43// C++ code for all of the loaded functions/hooks/event handlers. Thus,
44// for example, "zeek -b -O gen-C++ foo.zeek" will generate C++ code for
45// all of the scripts loaded in "bare" mode, plus those for foo.zeek; and
46// without the "-b" for all of the default scripts plus those in foo.zeek.
47//
48// "-O report-C++" reports on which compiled functions will/won't be used
49// (including ones that are available but not relevant to the scripts loaded
50// on the command line). This can be useful when debugging to make sure
51// that you're indeed running compiled code when you expect to be.
52//
53// We partition the methods of the compiler into a number of groups,
54// the definitions of each having their own source file:
55//
56// Driver Drives the overall compilation process.
57//
58// Vars Management of C++ variables relating to local/global
59// script variables.
60//
61// DeclFunc Generating declarations of C++ subclasses and
62// functions.
63//
64// GenFunc Generating the bodies of script functions.
65//
66// Consts Dealing with Zeek script constants. Depending
67// on their type, these are represented either
68// directly in C++, or using C++ variables that
69// are constructed at run-time.
70//
71// Stmts Generating code for Zeek statements.
72//
73// Exprs Generating code for Zeek expressions.
74//
75// Types Management of (1) C++ types used in generated code,
76// and (2) C++ variables that hold Zeek script types,
77// generated at run-time.
78//
79// Attrs Management of Zeek type attributes, some of which
80// must be generated at run-time.
81//
82// Inits Management of initializing the run-time
83// variables needed by the compiled code.
84//
85// Emit Low-level code generation.
86//
87// Of these, Inits is the most subtle and complex. There are two major
88// challenges in creating run-time values (such as Zeek types and constants).
89//
90// First, generating individual code for creating each of these winds up
91// incurring unacceptable compile times (for example, clang compiling all
92// of the base scripts with optimization takes many hours on a high-end
93// laptop). As a result, we employ a table-driven approach that compiles
94// much faster (though still taking many minutes on the same high-end laptop,
95// running about 40x faster however).
96//
97// Second, initializations frequently rely upon *other* initializations
98// having occurred first. For example, a global might need a record type
99// to be defined; one of the record's fields is a table; that table contains
100// another record; one of that other record's fields is the original record
101// (recursion); another field has an &default expression that requires the
102// compiler to generate a helper function to construct the expression
103// dynamically; and that helper function might in turn refer to other types
104// that require initialization. What's required is a framework for ensuring
105// that everything occurs in the proper order.
106//
107// The logic for dealing with these complexities is isolated into several
108// sets of classes. InitsInfo.{h,cc} provides the classes related to tracking
109// how to generate initializations in the proper order. RuntimeInits.{h,cc}
110// provides the classes used when initialization generated code in order
111// to instantiate all of the necessary values. See those files for discussions
112// on how they address the points framed above.
113//
114// In declaring the CPPCompiler class, we group methods in accordance with
115// those listed above, locating member variables with the group most relevant
116// for their usage. However, keep in mind that many member variables are
117// used by multiple groups, which is why we haven't created distinct
118// per-group classes. In addition, we make a number of methods public
119// in order to avoid the need for numerous "friend" declarations to allow
120// associated classes (like those for initialization) access to a the
121// necessary compiler methods.
