3// BaseList.h --
4// Interface for class BaseList, current implementation is as an
5// array of ent's. This implementation was chosen to optimize
6// getting to the ent's rather than inserting and deleting.
7// Also pairs of append's and get's act like push's and pop's
8// and are very efficient. The only really expensive operations
9// are inserting (but not appending), which requires pushing every
10// element up, and resizing the list, which involves getting new space
11// and moving the data. Resizing occurs automatically when inserting
12// more elements than the list can currently hold. Automatic
13// resizing is done by growing by GROWTH_FACTOR at a time and
14// always increases the size of the list. Resizing to zero
15// (or to less than the current value of num_entries)
16// will decrease the size of the list to the current number of
17// elements. Resize returns the new max_entries.
18//
19// Entries must be either a pointer to the data or nonzero data with
20// sizeof(data) <= sizeof(void*).

208 T remove_nth(int n) // delete nth entry from list
209 {

215 // the list in the same order when removing an element. Just swap the last

299 // This could essentially be an std::vector if we wanted. Some
300 // reasons to maybe not refactor to use std::vector ?
301 //
302 // - Harder to use a custom growth factor. Also, the growth
303 // factor would be implementation-specific, taking some control over
304 // performance out of our hands.
305 //
306 // - It won't ever take advantage of realloc's occasional ability to
307 // grow in-place.
308 //
309 // - Combine above point this with lack of control of growth
310 // factor means the common choice of 2x growth factor causes
311 // a growth pattern that crawls forward in memory with no possible
312 // re-use of previous chunks (the new capacity is always larger than
313 // all previously allocated chunks combined). This point and
314 // whether 2x is empirically an issue still seems debated (at least
315 // GCC seems to stand by 2x as empirically better).
316 //
317 // - Sketchy shrinking behavior: standard says that requests to
318 // shrink are non-binding (it's expected implementations heed, but
319 // still not great to have no guarantee). Also, it would not take
320 // advantage of realloc's ability to contract in-place, it would
321 // allocate-and-copy.
