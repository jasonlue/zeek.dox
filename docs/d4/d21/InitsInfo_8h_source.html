3// Classes for tracking information for initializing C++ values used by the
4// generated code.
5
6// Initialization is probably the most complex part of the entire compiler,
7// as there are a lot of considerations. There are two basic parts: (1) the
8// generation of C++ code for doing run-time initialization, which is covered
9// by the classes in this file, and (2) the execution of that code to do the
10// actual initialization, which is covered by the classes in RuntimeInits.h.
11//
12// There are two fundamental types of initialization, those that create values
13// (such as Zeek Type and Val objects) that will be used during the execution
14// of compiled scripts, and those that perform actions such as registering
15// the presence of a global or a lambda. In addition, for the former (values
16// used at run-time), some are grouped together into vectors, with the compiled
17// code using a hardwired index to get to a particular value; and some have
18// standalone globals (for example, one for each BiF that a compiled script
19// may call).
20//
21// For each of these types of initialization, our general approach is to a
22// class that manages a single instance of that type, and an an object that
23// manages all of those instances collectively. The latter object will, for
24// example, attend to determining the offset into the run-time vector associated
25// with a particular initialized value.
26//
27// An additional complexity is that often the initialization of a particular
28// value will depend on *other* values having already been initialized. For
29// example, a record type might have a field that is a table, and thus the
30// type corresponding to the table needs to be available before we can create
31// the record type. However, the table might have a set of attributes
32// associated with it, which have to be initialized before we can create the
33// table type, those in turn requiring the initialization of each of the
34// individual attributes in the set. One of those attributes might specify
35// a &default function for the table, requiring initializing *that* value
36// (not just the type, but also a way to refer to the particular instance of
37// the function) before initializing the attribute, etc. Worse, record types
38// can be *indirectly recursive*, which requires first initializing a "stub"
39// for the record type before doing the final initialization.
40//
41// The general strategy for dealing with all of these dependencies is to
42// compute for each initialization its "cohort". An initialization that
43// doesn't depend on any others is in cohort 0. An initialization X that
44// depends on an initialization Y will have cohort(X) = cohort(Y) + 1; or,
45// in general, one more than the highest cohort of any initialization it
46// depends on. (We cut a corner in that, due to how initialization information
47// is constructed, if X and Y are for the same type of object then we can
48// safely use cohort(X) = cohort(Y).) We then execute run-time initialization
49// in waves, one cohort at a time.
50//
51// Because C++ compilers can struggle when trying to optimize large quantities
52// of code - clang in particular could take many CPU *hours* back when our
53// compiler just generated C++ code snippets for each initialization - rather
54// than producing code that directly executes each given initialization, we
55// instead employ a table-driven approach. The C++ initializers for the
56// tables contain simple values - often just vectors of integers - that compile
57// quickly. At run-time we then spin through the elements of the tables (one
58// cohort at a time) to obtain the information needed to initialize any given
59// item.
60//
61// Many forms of initialization are specified in terms of indices into globals
62// that hold items of various types. Thus, the most common initialization
63// information is a vector of integers/indices. These data structures can
64// be recursive, too, namely we sometimes associate an index with a vector
65// of integers/indices and then we can track multiple such vectors using
66// another vector of integers/indices.

656// Much of the table-driven initialization is based on vectors of indices,
657// which we represent as vectors of int's, where each int is used to index a
658// global C++ vector. This class manages such vectors. In particular, it
659// reduces a given vector-of-indices to a single value, itself an index, that
660// can be used at run-time to retrieve a reference to the original vector.
661//
662// Note that the notion recurses: if we have several vector-of-indices, we can
663// reduce each to an index, and then take the resulting vector-of-meta-indices
664// and reduce it further to an index. Doing so allows us to concisely refer
665// to a potentially large, deep set of indices using a single value - such as
666// for CPP_InitsInfo's "offset_set" member variable.
667
668class IndicesManager
669 {
670public:
671 IndicesManager() { }
672
673 // Adds a new vector-of-indices to the collection we're tracking,
674 // returning the offset that will be associated with it at run-time.
675 int AddIndices(std::vector<int> indices)
676 {
677 int n = indices_set.size();
678 indices_set.emplace_back(std::move(indices));
679 return n;
680 }
681
682 // Generates the initializations used to construct the managed
683 // vectors at run-time.
684 void Generate(CPPCompile* c);
685
686private:
687 // Each vector-of-indices being tracked. We could obtain some
688 // space and time savings by recognizing duplicate vectors
689 // (for example, empty vectors are very common), but as long
690 // as the code compiles and executes without undue overhead,
691 // this doesn't appear necessary.
692 std::vector<std::vector<int>> indices_set;
693 };
